# 問題の追加手順

`title` / `question` / `answer` が与えられたとき、以下の手順で `puzzles.json` に追加する。

---

## 1. 重複チェック

`puzzles/puzzles.json` を読み、既存の問題と `question` が一致するものがないか確認する。  
一致する問題があれば、その旨を伝えて追加を中止する（`title` が違っても `question` が同じなら重複とみなす）。

---

## 2. id の採番

既存の最大 `id` + 1 を新しい `id` とする。

---

## 3. answer の記述チェック

elements を設計する前に、`answer` の文章が以下の条件を満たしているか確認する。

### 【厳守】登場人物の属性を明記する

answer に文化的慣習・特定の行為・固有の物が登場する場合、その担い手の属性（年齢・性別・職業など）が文化的常識から推測できるときは、**answer にも明記する**。

**NGの例（属性が未明記）:**
- （省略）

**OKの例（属性を明記）:**
- （省略）

**チェック観点:**
- 年齢が推測できる慣習・行動が含まれるか
- 職業が推測できる行為が含まれるか
- 性別が推測できる設定が含まれるか

属性が不明確な場合は、answer の修正をユーザーに提案してから次のステップに進む。

---

## 4. elements の設計

`answer` に含まれていて `question` には含まれていない情報を要素として抽出し、依存グラフを構築する。

### 要素数の目安

| 問題の複雑さ | 要素数 |
|---|---|
| シンプル（1つのキー情報で解ける） | 2 |
| 中程度 | 3〜4 |
| 複雑（複数の背景情報が必要） | 5〜8 |

上限は **8要素**。

### 依存グラフの設計原則

- **entry point**（`depends_on: []`）は 1〜3 個。ユーザーが question の情報だけから発想できる出発点。
- **中間要素**は entry point を具体化・深掘りしたもの。
- **core 要素**（核心・答えの鍵）は複数の要素が合流した末端に置く。

よくある構造パターン：

```
線形:        A → B
分岐:        A → B
                 → C
合流:        A ──┐
             B ──→ D（core）
             C ──┘
複数入口:    A ──────────────┐
             B → B1 → B2 ──→ E（core）
             C → C1 → C2 ──┘
```

### 各フィールドの書き方

| フィールド | 内容 |
|---|---|
| `id` | アルファベット識別子。entry pointは `A` `B` `C`、派生は `B1` `B2` `C1`、core は `E` |
| `label` | 設計用の短いラベル（プレイヤーには見えない） |
| `hint` | プレイヤー向けの文言。**entry pointは方向性のみ、深い要素ほど具体的に**。ただし答えを直接明かさない |
| `depends_on` | 前提となる要素の `id` リスト。entry pointは `[]` |

### hint の書き方ガイドライン

- entry point: 「〇〇がポイント」「男には〜な背景がある」など、何に注目すべきかだけを示す
- 中間: カテゴリや状況を絞り込む
- core: 「〜が、男にとって耐えられない事実だった」など、ほぼ答えに近いが直接的ではない表現

### 【厳守】hint に答えを直接書かない

answerの核心となる固有名詞・行為・状況をそのままhintに書いてはいけない。

**NGの例:**
- （省略）

**OKの例（同じ要素を間接的に表現）:**
- （省略）

**判定基準:** hintを読んだだけで答えが即座にわかってしまう場合はNG。「その方向で考えれば解けるかもしれない」という程度の情報量に留める。

### ヒント番号

`elements` 配列の 1-indexed 位置がヒント番号になる（実装依存）。  
依存グラフの entry point を先に並べ、深い要素を後に配置すること。

## 6. 追加内容の確認表示

JSON追加前に、設計したelementsを以下の形式で表示し確認を求める。  
**メインの設計（N要素）に加え、N-2・N-1・N+1 要素に調整したバリエーションも同時に提示する**（ただし最小2要素、最大8要素の範囲内）。

```
【id: X タイトル】elements設計

■ N要素（メイン案）
| # | id | label | hint | depends_on |
|---|---|---|---|---|
| 1 | A | ... | ... | [] |
| 2 | B | ... | ... | ["A"] |
...

■ N-1要素
...

■ N-2要素
...

■ N+1要素
...
```

---

## 7. JSON への追加

上記の設計をもとに、以下の形式で `puzzles.json` に追加する。

```json
{
  "id": <次のid>,
  "title": "<タイトル>",
  "question": "<問題文>",
  "answer": "<答え>",
  "elements": [
    {
      "id": "A",
      "label": "<設計用ラベル>",
      "hint": "<プレイヤー向けヒント>",
      "depends_on": []
    },
    ...
  ]
}
```
